# দিন ৫: Pull Requests ও Code Review

## Pull Request (PR) কী?

**Pull Request** হল একটি ফিচার যেখানে আপনি আপনার কোড পরিবর্তনগুলো **মেইন ব্রাঞ্চে মার্জ করার আগে** টিমকে রিভিউ করতে দেন।

**সোজা ভাষায়:** আপনি বলছেন — "আমি এই পরিবর্তনগুলো করেছি, দয়া করে দেখুন এবং main-এ মার্জ করুন।"

## কেন PR গুরুত্বপূর্ণ?

✅ কোড রিভিউ ও কোয়ালিটি নিশ্চিত করে  
✅ বাগ আগেই ধরা যায়  
✅ টিম সবাই জানে কী পরিবর্তন হচ্ছে  
✅ ডকুমেন্টেশন হিসেবে কাজ করে  
✅ Branching-এর সাথে মিলিয়ে ব্যবহার করা হয়

## Pull Request তৈরি করার ধাপ

### ধাপ ১: ফিচার ব্রাঞ্চ তৈরি করুন
```bash
git checkout -b feature/clean-sales-data
```

### ধাপ ২: কাজ করুন ও commit করুন
```bash
# ডাটা ক্লিনিং কোড লিখুন
git add clean_data.py
git commit -m "সেলস ডাটা ক্লিনার ফাংশন যোগ করা হয়েছে"
```

### ধাপ ৩: ব্রাঞ্চ Push করুন
```bash
git push -u origin feature/clean-sales-data
```

### ধাপ ৪: GitHub-এ PR তৈরি করুন

1. GitHub-এ আপনার রিপোজিটরিতে যান
2. "Pull requests" ট্যাবে ক্লিক করুন
3. "New pull request" বাটনে ক্লিক করুন
4. **base:** main ← **compare:** feature/clean-sales-data
5. টাইটেল ও ডেসক্রিপশন লিখুন (বাংলায় লিখতে পারেন)
6. "Create pull request" ক্লিক করুন

### ধাপ ৫: কোড রিভিউ

রিভিউয়াররা কমেন্ট করতে পারে, পরিবর্তন চাইতে পারে:
- ✅ **Approve** — মার্জ করার জন্য অনুমোদন
- ❌ **Request changes** — কিছু পরিবর্তন চান
- 💬 **Comment** — শুধু মতামত

### ধাপ ৬: PR মার্জ করা

রিভিউ শেষে "Merge pull request" বাটনে ক্লিক করুন।

## ভালো PR-এর বৈশিষ্ট্য

### টাইটেল ও ডেসক্রিপশন (বাংলায় লিখুন):
```
ডাটা ক্লিনিং ফাংশন — NULL ভ্যালু হ্যান্ডলিং

এই PR-এ কী যোগ করা হয়েছে:
- sales_data.csv-তে NULL ভ্যালু হ্যান্ডলিং
- আউটলায়ার ডিটেকশন ফাংশন
- ডাটা টাইপ কনভার্সন ফিক্স

কী টেস্ট করা হয়েছে:
- ১০০০ রো-র ডাটাসেটে টেস্ট করা হয়েছে
- সব টেস্ট পাস করেছে
```

### PR-এর আকার:
- **ছোট PR** (১০-৫০ লাইন) → সহজে রিভিউ হয়
- **বড় PR** (২০০+ লাইন) → রিভিউ করতে সময় লাগে, বেশি ভুল থাকে

### Data Analyst-এর জন্য PR আইডিয়া:
```
PR টাইটেল: "মাসিক সেলস রিপোর্ট — Q4 2024 ডাটা যোগ করা হয়েছে"
PR টাইটেল: "ভিজুয়ালাইজেশন — নতুন বার চার্ট টেমপ্লেট"
PR টাইটেল: "বাগ ফিক্স — ডাটা ফিল্টার ঠিক করা হয়েছে"
```

## Code Review করার সময় কী দেখবেন?

🔍 **লজিক:** এনালাইসিস ঠিক আছে কিনা  
🔍 **পারফরম্যান্স:** কোড কি ধীরগতির?  
🔍 **রিডেবিলিটি:** কোড কি সহজে বোঝা যায়?  
🔍 **কনসিস্টেন্সি:** প্রজেক্টের স্টাইলের সাথে মিলছে?  
🔍 **টেস্ট:** কীভাবে নিশ্চিত যে কাজ করবে?

## PR মার্জ অপশন

GitHub তিন ধরনের মার্জ দেয়:

1. **Create a merge commit** — সব commit ইতিহাস রাখে
2. **Squash and merge** — সব commit একটিতে পরিণত করে (পরিষ্কার ইতিহাস)
3. **Rebase and merge** — commit ইতিহাস লিনিয়ার রাখে

Data Analyst-দের জন্য **Squash and merge** সাধারণত ভালো — পরিষ্কার ইতিহাস থাকে।
